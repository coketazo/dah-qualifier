- 채점(100점): 공격 시나리오 30 / 방어 전략 25 / AI 에이전트 25 / 팀 역량 10 / 문서 완성도 10. 0.5점 단위, 평가위원 평균.

- 본선 환경(사이트 확인): 클라우드 UAV/UGV 가상환경 + 위성망 모사 인프라, 팀이 에이전트를 클라우드에 배포, **AI 공격 vs AI 방어 자율 실시간 공방**. 채점 `(공격점수+방어점수)×가용성계수`, 가용성 0~100, SLA 실패 시 감소("공격 성공해도 시스템 다운되면 감점").
- 진실성(제11조②): 생성형 AI 활용 시 보고서에 도구·활용범위 명시 권장. **단, 제한적·보수적으로 기재** — "아키텍처 개념 학습·도메인 지식 학습·에이전트 및 표적 서버 구현 보조"에 활용했다고만. 전면적으로 AI가 다 했다는 인상은 감점 리스크이므로 독창적 설계·판단은 팀의 것으로 서술.

## 1. 전략 방향

- 하나의 **연결된 킬체인 스토리**로 ④⑤⑥을 관통(산발적 취약점 나열 금지).
- 무게중심 항목(공격·에이전트)에 힘을 집중. 방어는 견고한 아키텍처 + 실동 데모로 뒷받침.
- 본선 정합성: **가용성-인지(availability-aware) 방어**를 핵심 차별점으로 제시.
- 리스크 관리: 레드/블루 둘 다 구현하되 **각 에이전트는 좁게**(핵심 동작만). 모의 표적은 최소·명확.

## 2. 시나리오 축 (확정)

**위성링크(BLOS) 경유 UAV 지상관제소(GCS)/C2 체계 침해 및 자율 대응.**

방산 특수성 앵커(취미드론급 소재 탈피):
- **BLOS 위성 C2**: 가시선 밖 위성 중계 지휘통제 링크(사이트 '위성망 모사'와 정합).
- **GPS-denied / 전자전(EW)**: 재밍·기만 하에서의 항법·링크 신뢰성.
- **군용 데이터링크 특성**: 명령 무결성·재전송·페일세이프.
- **AI-native 위협**: 온보드 표적탐지/자율판단 AI에 대한 교란(적대적 예제) — 설계에 정식 포함.

## 3. 공격 시나리오 (보고서 §4, 30점) — 킬체인

의도적으로 취약하게 구성한 **모의 GCS/C2**에 대해 자율 레드 에이전트가 수행:

1. **정찰(Recon)**: 위성링크 경유 노출된 GCS 웹 대시보드 / REST 명령 API / 텔레메트리 스트림(WebSocket) 식별, 엔드포인트·인증방식 매핑.
2. **초기 침투(Access)**: GCS 웹/API 인증 취약점(약한 세션 토큰·미검증 API 키·CSRF)으로 오퍼레이터 권한 획득.
3. **명령 주입(Command Injection, 핵심)**: 명령 채널 무결성 부재를 악용해 위조 명령/미션(웨이포인트 재작성, RTL·페일세이프 우회) 주입 → 기체 침로 변경.
4. **기만(Deception)**: 위조 텔레메트리 + GPS 스푸핑으로 오퍼레이터 화면엔 정상처럼 표시하며 실제 기체를 오유도.
5. **지속·확산(Persistence/Lateral)**: 스웜(다중 UAV) 환경에서 C2 노드 장악 후 인접 노드로 확산.
- **AI-native 서브벡터**: 온보드 표적탐지 모델에 대한 적대적 입력으로 오탐/미탐 유도(설계+경량 데모).

각 단계는 실제 근거·문헌과 연결(§10 참고문헌): MAVLink 무인증/서명, GPS 스푸핑 실증(Humphreys 2012), RQ-170(2011), GCS 멀웨어(Creech AFB 2011), 적대적 예제(Goodfellow 2015) 등.

## 4. 방어 아키텍처 (보고서 §5, 25점) — 탐지·차단·복구 + 가용성 보존

공격 각 단계에 1:1 대응:

| 공격 단계 | 탐지 | 차단 | 복구/가용성 |
|---|---|---|---|
| 정찰 | 비정상 스캔·열거 트래픽 | rate limit·핑거프린팅 차단 | 정상 세션 유지 |
| 침투 | 인증 이상(세션/토큰) | 세션 격리·MFA·토큰 폐기 | 오퍼레이터 재인증 |
| 명령 주입 | 명령 화이트리스트·물리모델 정합성(속도/고도/위치 급변) | 위조 명령 드롭 | 정상 명령 경로 유지 |
| 기만 | 텔레메트리·다중신호 정합성, GPS 스푸핑 탐지 | 오염 신호 격리 | 안전 페일세이프 복귀 |
| 확산 | 노드 간 이상 상관 | 채널 재키잉·세그멘테이션 | 스웜 부분 격리로 임무 지속 |

**가용성 원칙**: 방어가 시스템을 셧다운하지 않는다. graceful degradation, 안전 페일세이프, 정상 명령 경로 보존. (본선 SLA 계수 정합)

## 5. AI 에이전트 아키텍처 (보고서 §6, 25점) — 레드+블루 (둘 다 구현)

### 5.1 레드 에이전트 (완전 구현, 중심축)
- 구조: LLM 오케스트레이터(ReAct: 관측→추론→행동) + 툴셋.
- 툴: `recon_scan`(엔드포인트 탐색), `auth_exploit`(인증우회), `command_inject`(위조 명령/미션 주입), `telemetry_spoof`(기만).
- 산출: 단계별 판단 로그(자율성 입증), 성공/실패 지표.

### 5.2 블루 에이전트 (구현, 좁게)
- 구조: 이벤트 스트림 관측 → **규칙기반 탐지(신뢰성 보장)** + LLM 트리아지/설명(혁신성) → 대응 액션 선택.
- 탐지 룰: 명령 화이트리스트, rate limit, 물리 정합성 검사, 서명 검증.
- 대응: 위조 명령 drop / 세션 격리 / 알림. **가용성 지표를 보상/판단 신호로 사용**(정상 트래픽 살림).
- 산출: 탐지 이벤트 로그, 차단 결과, 가용성(SLA) 추이.

### 5.3 대결 무대
- 레드·블루가 **같은 모의 GCS**에서 실시간 상호작용. 대시보드에 공방 타임라인 + 가용성 게이지 표시.

### 5.4 기술 스택 (웹개발 강점 활용)
- Python 3.11+ / **FastAPI**(모의 GCS: 웹 대시보드 + REST 명령 API), **WebSocket**(텔레메트리 스트림).
- 에이전트: Python, LLM = Claude API(ReAct 루프). 블루 룰엔진은 순수 Python.
- 프론트: 간단 HTML/JS(또는 경량 React) 대시보드로 데모 시각화.
- 재현: `Dockerfile` / `docker-compose`, `requirements.txt`.

## 6. 모의 표적 환경 `mock-gcs` (컴포넌트 스펙)

- 역할: 레드가 공격하고 블루가 지키는 **단일 무대**. 실제 무기체계 미사용(합법·재현가능).
- 인터페이스:
  - `GET /api/telemetry` + `ws://…/telemetry`: 기체 상태(위치/고도/속도/모드) 스트림.
  - `POST /api/command`: 명령 주입 지점(의도적 무인증 취약점 토글 가능).
  - `GET /` : 오퍼레이터 대시보드(지도/상태/명령 이력).
- 상태 모델: 단순 비행 시뮬레이터(웨이포인트 추종), 가용성/SLA 카운터 내장.
- 취약점 스위치: `SECURE=false/true`로 방어 on/off 비교 데모.

## 7. 컴포넌트 경계 (독립·테스트 가능)

```
src/
  mock_gcs/      # 표적: FastAPI 앱 + 시뮬레이터 (트랙 A, 웹개발자)
  red_agent/     # 공격 에이전트 (트랙 B + Claude 골격)
  blue_agent/    # 방어 에이전트 (트랙 B + Claude 골격)
  common/        # MAVLink-lite 메시지·이벤트 스키마·SLA 계산
  demo/          # 공방 시나리오 러너 + 로그/스크린샷 생성
```
- 각 유닛은 스키마(common)로만 통신. 개별 실행·테스트 가능.

## 9. 작업 계획 (~1.5일, 2인 + Claude)

- 트랙 A(웹개발자 본인): `mock_gcs` 표적 + 대시보드.
- 트랙 B(친구): `red_agent`/`blue_agent` 채우기·테스트 (Claude가 골격 제공).
- Claude(병렬): 보고서 전 섹션 초안 집필 + 다이어그램(텍스트/mermaid) + 참고문헌 정리 + 코드 골격.
- 순서: (1) 공통 스키마+mock_gcs 최소 → (2) red_agent 관통(정찰→주입) → (3) blue_agent 탐지·차단 → (4) 데모 러너·스크린샷 → (5) 보고서 확정·PDF 변환 → (6) ZIP 업로드·링크.

## 11. 리스크 & 대응

- 과범위(레드+블루+데모) → 각 에이전트 좁게, Claude가 골격 선제작, 데모는 스크립트화.
- 무경험(에이전트/공방) → ReAct 최소 구현, 블루는 규칙기반 우선.
- AI-native ML 부담 → 학습 없이 사전학습/규칙 기반 경량 데모 또는 설계 중심.
- 시간 부족 시 우선순위: 보고서 완성 > 레드 데모 > 블루 데모 > 영상.
